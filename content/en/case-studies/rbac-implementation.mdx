<Callout type="problem">
UFT has 13 role-scoped portals across a domain hierarchy (Federation → League → Club → Team → Player) — federation presidents, league presidents, club admins, coaches, team leaders, tournament organizers, scorekeepers, and players all need different, narrower capabilities on the same underlying data. Early on, roles got seeded inconsistently across two separate seeders plus ad-hoc strings used directly in `@Roles(...)` decorators, so the same conceptual role ended up with multiple literal name variants in the database.
</Callout>

<Callout type="solution">
Instead of a risky one-time migration to rename every role consistently across live data, I built a role-alias table that maps known variants (`"Organizador"`, `"Tournament Organizer"` → `"Organizador de Torneo"`, etc.) to a canonical name, so a `@Roles(...)` check passes regardless of which variant a given user was actually seeded with — grouping only variants of the *same* privilege level, never merging across levels. Access control itself runs through a single guard checking both role-based (`@Roles`) and permission-based (`@RequirePermissions`) metadata: if a route declares neither, it's open to any authenticated user; if it declares either, the request must satisfy it. "Super Administrador" is a hardcoded bypass, deliberately — safe specifically because that role can't be self-assigned (a privilege-escalation path that let a user grant themselves admin was found and closed). The safe default is explicit: a required check that's simply absent never grants access on its own.
</Callout>

<Callout type="tradeoffs">
The alias table avoided a mass-rename across live data, but it's a permanent piece of indirection now — and it coexists with two parallel role/permission entity definitions (one under `auth/`, one under a separate `rbac/` module) that were never fully unified. Real, acknowledged debt: pragmatic in the moment, not free.
</Callout>

<Callout type="lessons">
A security review run against this system in July 2026 found and fixed four concrete issues: the privilege-escalation path noted above, a guard that wasn't actually enforcing what it appeared to, an open seed endpoint, and passwords showing up in logs — each with a regression test added afterward. An IDOR review of file-upload endpoints (avatar, player photo, banner, venue photo) found they weren't checking that the uploading user actually owned the resource being modified; that's now validated against the authenticated user, backed by 11 regression tests. Two other upload endpoints (generic image and document) are explicitly still open and tracked, not silently left unfixed. The real lesson: a permission system that accumulates role-name variants across multiple seeders will eventually need a compatibility layer or a breaking migration — the alias table was the lower-risk call given live data, but the two parallel entity definitions are the real fix still owed. And a security review is only as useful as what happens after it — finding four issues mattered less than pairing each with a regression test, and honestly tracking what's still open.
</Callout>
