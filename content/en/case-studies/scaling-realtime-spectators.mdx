<Callout type="problem">
UFT runs on a single DigitalOcean droplet hosting Postgres, the API, NATS, Redis, the social feed service, and live streaming, all together. Spectator traffic around live matches is inherently spikier than steady admin usage, and rather than guess how the current setup holds up, the goal was to actually measure it — connections, join latency, error rate, and where deploys themselves lose time.
</Callout>

<Callout type="solution">
An initial load-testing attempt used a generic tool (k6) that connected to the wrong Socket.io namespace, produced zero visible backend activity, and looked like a total failure. Rather than trust that signal, I wrote load-test scripts in Node using the exact same `socket.io-client` library the real frontend uses, so test traffic is indistinguishable from real traffic at the protocol level. Built tiered, purpose-specific scenarios instead of one generic test: single-match, 4-simultaneous-match, and an "all live matches dashboard" scenario, each scaling through light/medium/heavy/stress tiers (10/25/50/100 spectators per match). Separately, deploy time itself was measured rather than assumed acceptable: a root-cause analysis broke a 20–45 minute deploy down phase by phase.
</Callout>

<Callout type="tradeoffs">
Hand-rolled load scripts take more upkeep than an off-the-shelf tool, but they're the only thing that told the truth here — the generic tool's false negative would have sent the investigation in the wrong direction. On the deploy side, the fix (stop deleting Docker's build cache and forcing a `--no-cache` frontend rebuild every deploy) is identified and estimated to save the largest chunk of the 20–45 minutes, but hasn't been applied yet — a real, open gap between "documented" and "done."
</Callout>

<Callout type="lessons">
Connection tests: 100% of spectators connected and joined their match room successfully, 0 errors. But average connection time was ~3.9 seconds and join-room time ~1.9 seconds — against the team's own benchmark (under 500ms as "ideal"), that's honestly labeled "high," not acceptable. Targeted concurrency tests separately validated 100+ spectators in a single room and 10+ simultaneous matches with clean room isolation, all passing. On deploys: the 20–45 minute figure broke down to roughly 12–25 minutes just for the frontend build, caused by deleting the previous image and disabling the build cache before every deploy. Correctness was validated before latency was — a defensible order, but "it works" and "it's fast" are genuinely different bars, and ~3.9s is the honest gap between them right now. The load-testing story is itself the bigger lesson: a general-purpose tool that doesn't match your actual client can produce a confident, wrong signal — testing with the real client library is what turned "seems broken" into "here's the actual number, and here's the target it's missing."
</Callout>
