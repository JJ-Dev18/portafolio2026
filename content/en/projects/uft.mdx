## Overview

UFT is a system for running Ultimate Frisbee tournaments: registrations, tournament automation, live scorekeeping, statistics, spirit-of-the-game scoring, a social feed, and live spectator streaming — used by organizers, scorekeepers, coaches, and spectators during real events, across 13 role-scoped portals covering the sport's own hierarchy (federation → league → club → team → player). It's been in production since March 19, 2025, currently tracking around 1,700 users through Google Analytics.

## Why I built it

Tournament scorekeeping is often manual and fragmented — spreadsheets, group chats, paper. That's slow to update and easy to get wrong, and it gives spectators nothing to follow along with in real time. UFT exists to make live tournament data — scores, standings, statistics — correct and immediately visible, for the people running the event and the people watching it, whether they're in the room or watching a live stream.

## Architecture

UFT is a small system, not a single API: a NestJS backend (32 feature modules) owns the tournament domain, a separate NestJS microservice owns the social feed with its own database and Redis cache, and NATS JetStream connects the two so the social feed can be down without touching live scorekeeping. A React web app, an Expo/React Native mobile app used in the field, and an nginx-rtmp live-streaming layer round out the system. The full breakdown — diagrams, module list, database design — lives in [Engineering](/engineering); this section stays focused on the product.

## Technical challenges

The hardest problems in building UFT so far:

- **Realtime scorekeeping** with Ultimate's own rules (breaks, Callahans, mixed-division gender ratios) enforced live, across scorekeeper and spectator clients at once — see [Designing realtime scorekeeping](/case-studies/realtime-scorekeeping).
- **Configurable tournament formats** — round robin, single elimination, groups-into-elimination, and custom ranking brackets — instead of hardcoding one format per tournament type — see [Optimizing tournament standings and bracket generation](/case-studies/tournament-standings-optimization).
- **Access control** across 13 role-scoped portals, including cleaning up role names that drifted across two different seeders — see [RBAC implementation](/case-studies/rbac-implementation).

## Engineering decisions

The event-driven layer (NATS JetStream) is the decision with the most downstream effect on the rest of the system — see [Event-driven architecture using NATS](/case-studies/event-driven-architecture-nats) for why it's there, and an honest account of which parts of that design are actually wired up today versus just declared.

## Current infrastructure

Docker Compose runs Postgres, the API, NATS, Redis, the social feed service, live streaming, and automated nightly backups (7-day retention), all on one DigitalOcean droplet running staging and production side by side. CI/CD via GitHub Actions gates every deploy on backend and frontend test suites passing, with commit-tag-based selective deploys.

## Performance

Measured, not assumed: a hand-written load test (using the same `socket.io-client` the frontend uses) found 100% connection/join success and 0 errors, but average connection time around 3.9 seconds — which the team's own benchmark rates "high" against a 500ms target. Separately, deploys were measured too: a self-authored analysis found they take 20–45 minutes, mostly from rebuilding the frontend without any build cache on every single deploy. Full numbers and the fix list in [Scaling realtime spectators](/case-studies/scaling-realtime-spectators).

## Scalability

Targeted concurrency tests validate 100+ spectators in one room and 10+ simultaneous matches with clean room isolation — the correctness story holds. The latency story doesn't yet: connection times are real headroom, not a solved problem, and the single-droplet setup hasn't been load-tested against a genuine spectator-scale spike. What's next is in [Scaling realtime spectators](/case-studies/scaling-realtime-spectators).
