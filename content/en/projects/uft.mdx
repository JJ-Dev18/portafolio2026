## Overview

UFT is a platform for running Ultimate Frisbee tournaments: registrations, tournament management, live scorekeeping, and real-time statistics, used by organizers and spectators during actual events. It's been in production since March 19, 2025, currently tracking around 1,700 users through Google Analytics.

## Why I built it

Tournament scorekeeping is often manual and fragmented — spreadsheets, group chats, paper. That's slow to update and easy to get wrong, and it gives spectators nothing to follow along with in real time. UFT exists to make live tournament data (scores, standings, statistics) correct and immediately visible, for the people running the event and the people watching it.

## Architecture

UFT is a NestJS API backed by PostgreSQL, with NATS JetStream as an event backbone connecting the parts of the system that need to react to state changes without blocking the request path. The full breakdown — system diagram, backend structure, database design, and deployment topology — lives in [Engineering](/engineering); this section stays focused on the product.

## Technical challenges

The three hardest problems in building UFT so far:

- **Realtime scorekeeping** with more than one person able to record a score for the same match — see [Designing realtime scorekeeping](/case-studies/realtime-scorekeeping).
- **Recomputing standings** fast enough that it doesn't compete with the live-scoring request path — see [Optimizing tournament standings](/case-studies/tournament-standings-optimization).
- **Access control** across organizers, scorekeepers, and admins acting on the same tournament — see [RBAC implementation](/case-studies/rbac-implementation).

## Engineering decisions

The event-driven layer (NATS JetStream) is the decision with the most downstream effect on the rest of the system — see [Event-driven architecture using NATS](/case-studies/event-driven-architecture-nats) for why it's there and what it actually carries.

## Current infrastructure

Dockerized services behind Nginx, deployed on a DigitalOcean VPS. Simple, deliberately — the trade-offs of that choice are covered honestly in [Scaling realtime spectators](/case-studies/scaling-realtime-spectators), including what isn't solved yet.

## Performance

No formal benchmarks published yet. The only production signal I have today is Google Analytics usage data (~1,700 tracked users) — real load-testing numbers are on the list, not invented for this page.

## Scalability

The current single-VPS setup is sized for today's traffic, not spectator-scale spikes around live matches. What would need to change first is covered in [Scaling realtime spectators](/case-studies/scaling-realtime-spectators).
